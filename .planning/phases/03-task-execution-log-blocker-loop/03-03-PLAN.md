# Plan 03 — Task Board Drawer and Detail UI

<objective>
Implement the Phase 3 frontend execution cockpit: real task cards on the project board, editable task drawer, work-log/blocker interactions, one-level subtasks, and directly accessible `/tasks/:taskId` detail page.
</objective>

<user_story>
As a project member, I want to scan real task cards, open a drawer to advance work, record logs and blockers, and open full task details when needed, so that daily execution stays fast and traceable.
</user_story>

<threat_model>
T-03-09: Frontend could expose unauthorized actions as available. Gate participant, Owner, and project-manager actions in UI while relying on backend enforcement.
T-03-10: Fake UI data could mislead demo reviewers. Render only API-backed tasks, logs, blockers, and counts.
T-03-11: Hidden form labels or color-only states could break accessibility. Use visible labels, text-bearing status chips, and focusable controls.
T-03-12: Phase 4 actions could appear before backend support exists. Do not expose acceptance pass/reject/review controls.
</threat_model>

<must_haves>
- UI-SPEC: Task card, task drawer, work-log form, subtask checklist, blocker history, and task detail page contracts are implemented.
- D-01..D-06: Drawer plus `/tasks/:taskId` detail route.
- D-29..D-35: Medium-density cards, blocker priority, light log state, and drawer-centered work-log creation.
- D-36..D-39: Work-log form disables future dates and treats code fields as optional text.
- No fake Phase 4/5 data appears.
</must_haves>
