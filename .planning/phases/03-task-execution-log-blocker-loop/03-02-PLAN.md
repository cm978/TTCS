# Plan 02 â€” Work Logs, Blockers, and Task APIs

<objective>
Expose Phase 3 backend behavior through `/api/v1` JSON APIs and implement work-log/blocker lifecycle rules that produce reliable evidence data for later acceptance gates, workbench queues, and reports.
</objective>

<user_story>
As a task participant, I want to write work logs and mark or resolve blockers from a task, so that daily execution and delivery risk are visible before acceptance.
</user_story>

<threat_model>
T-03-05: Users could write logs on tasks they do not participate in. Require task participant, Owner, or project-manager authorization before writes.
T-03-06: Future-dated or invalid-hour logs could corrupt reporting. Validate date and hours server-side.
T-03-07: Blocker state could drift from blocker logs. Recompute task-level `is_blocked` and summary after every blocker create/resolve/update/soft-delete.
T-03-08: API responses could expose tasks outside the actor's project visibility. All task board/detail/list endpoints must pass task/project visibility checks.
</threat_model>

<must_haves>
- WORK-01: Task participants can create work logs with date, hours, content, and work type.
- WORK-02: Work-log hours are minimum `0.5` and maximum `24`.
- WORK-03: Commit hash, branch name, and repository URL are optional plain text fields.
- WORK-04: Members can mark blocker with reason at least 10 characters.
- WORK-05: Unresolved blockers make the task ineligible for future acceptance submission.
- WORK-06: Blocker data is available to Owner/project manager and future standup/report sources.
- D-21..D-28: Work-log driven blocker lifecycle is authoritative.
</must_haves>
