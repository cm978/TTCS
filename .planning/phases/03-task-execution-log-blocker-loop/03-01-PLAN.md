# Plan 01 — Task Domain Foundation

<objective>
Create the backend task-domain foundation for Phase 3: task tables, participant rules, one-level subtasks, dependencies, status guardrails, objective progress calculation, soft deletion, and tests.
</objective>

<user_story>
As a project member, I want to create and manage real tasks on a project board with Owner, participants, type, priority, due date, labels, subtasks, and dependencies, so that the board represents actual work instead of empty columns.
</user_story>

<threat_model>
T-03-01: Broken access control could let non-members create or edit project tasks. Enforce project membership, Owner, and project-manager checks in backend services.
T-03-02: Client-forged status or progress could fake task completion. Centralize legal transitions and compute progress from subtasks only.
T-03-03: Participant removal could erase evidence. Keep task history independent from current participant membership and never cascade-delete work evidence.
T-03-04: Cyclic dependencies could deadlock planning. Reject same-project dependency cycles in service tests.
</threat_model>

<must_haves>
- TASK-01: Tasks include title, description, task type, priority, due date, Owner, participants, board column/status, and labels.
- TASK-02: Backend enforces who may edit task basics.
- TASK-03: Project managers can soft-delete tasks; physical deletion is not used.
- TASK-04: Task types are exactly `GENERAL`, `DOCUMENT`, and `CODE`.
- TASK-05: Every task has an Owner and no more than 5 participants; Owner is auto-added and counts toward the limit.
- TASK-06: Phase 3 implements one-level checklist subtasks and objective progress.
- TASK-07: Same-project dependencies are supported and cycles are rejected.
- TASK-08: Task status follows legal guardrails and cannot jump directly to `DONE`.
</must_haves>
