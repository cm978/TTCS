# Plan 05 — UAT Task Execution UX Fixes

<objective>
Close Phase 3 UAT gaps in the frontend task execution experience: create tasks only after user confirmation, make task edits visibly save and refresh, expose due date editing, make subtask completion reliable, and move work-log/blocker entry behind explicit actions with clear validation feedback.
</objective>

<user_story>
As a project member, I want task creation, editing, subtasks, work logs, and blocker actions to behave like deliberate confirmed operations, so that I can trust the board reflects my real work instead of guessing whether a click saved.
</user_story>

<scope>
Fix Phase 3 task execution UX and client behavior only. Do not add Phase 4 acceptance review, notifications, reports, real Git sync, or AI review.
</scope>

<tasks>

<task id="03-05-01" title="Add confirmed task creation flow">
<read_first>
- `.planning/phases/03-task-execution-log-blocker-loop/03-UAT.md`
- `frontend/src/views/ProjectBoardView.vue`
- `frontend/src/components/task/TaskDrawer.vue`
- `frontend/src/stores/task.ts`
- `frontend/src/types/task.ts`
</read_first>
<action>
- Add a dedicated task creation drawer or modal, preferably `frontend/src/components/task/CreateTaskDrawer.vue`.
- Open the create UI from `创建任务` instead of immediately calling `taskStore.createTask`.
- Require a task title before creation and expose description, task type, priority, and due date before confirmation.
- Call `taskStore.createTask` only after the user confirms.
- After successful creation, refresh project tasks, open the created task detail drawer, and show clear success/error feedback.
</action>
<acceptance_criteria>
- Clicking `创建任务` does not create a card until the user submits the create form.
- Newly created tasks appear in the `待办` column only after the API call succeeds.
- Create validation prevents empty/default accidental tasks.
- Phase 4/5/v2 controls remain absent.
</acceptance_criteria>
</task>

<task id="03-05-03" title="Make subtask completion reliable and visible">

<task id="03-05-04" title="Move work-log entry behind explicit action with validation feedback">
<read_first>
- `frontend/src/components/task/TaskDrawer.vue`
- `frontend/src/components/task/WorkLogForm.vue`
- `frontend/src/stores/task.ts`
- `frontend/src/views/TaskDetailView.vue`
</read_first>
<action>
- Replace the always-visible work-log form with a `记录工作日志` action that opens a focused form section/modal.
- Keep fields for work date, hours, work type, content, optional commit hash, branch name, and repository URL.
- Show validation errors for future date, invalid hours, missing content, and blocked reason shorter than 10 characters.
- After save, close or reset the form, refresh work-log history, and show success/error feedback.
- Confirm optional code fields stay plain text and do not trigger live Git integration.
</action>
<acceptance_criteria>
- Work-log form appears only after the user chooses to record a log.
- Valid logs save and appear in task history.
- Invalid values show visible feedback.
- No real Git sync is triggered.
</acceptance_criteria>
</task>
