# Plan 02 — Team Project API Surface

<objective>
Expose Phase 2 team, invitation, project, project member, and board operations through protected `/api/v1` REST endpoints backed by the domain services from Plan 01.
</objective>

<user_story>
As an authenticated TTCS user, I want API endpoints for teams, invitations, projects, members, and boards, so that the frontend can create and manage a real collaboration workspace.
</user_story>

<threat_model>
T-02-API-AUTH: Missing current-user dependencies could expose team or project data. Every endpoint depends on `get_current_user`.
T-02-API-AUTHZ: Routers could bypass service-level permission checks. Route handlers must delegate mutations to `TeamService` and `ProjectService`.
T-02-API-DATA: API responses could leak unrelated team/project records. List/detail endpoints must filter by current user visibility.
</threat_model>

<must_haves>
- D-01..D-09: Team invitation API supports demo-first pending invitation flow.
- D-10..D-18: API preserves role boundaries and team-member-only project membership.
- D-26, D-29, D-30: Project board API returns persisted fixed columns.
- D-37..D-40: API exposes backend-enforced final-admin/final-manager errors.
</must_haves>
