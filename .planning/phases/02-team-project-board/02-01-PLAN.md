# Plan 01 — Team Project Domain Foundation

<objective>
Create the Phase 2 backend domain foundation: team/project models, roles, invitation and board enums, Alembic migration, shared permission helpers, and service-level invariants.
</objective>

<user_story>
As a TTCS user, I want teams, invitations, projects, roles, and default board columns to exist as real backend records, so that the UI and later task phases can build on stable collaboration data.
</user_story>

<threat_model>
T-02-01: Broken access control could let non-admins manage teams or non-managers manage projects. Centralize permission helpers and test negative paths.
T-02-02: Role changes could orphan a team or project with no administrator/manager. Enforce final-admin and final-manager rules in services.
T-02-03: Invitation hijacking or duplicate active invitations could add the wrong user. Normalize email, match logged-in user email, and block duplicate pending invites.
T-02-04: Project creation could partially write project data without board columns. Use one transaction for project, creator membership, and five columns.
</threat_model>

<must_haves>
- D-01..D-09: Implement demo-first email invitations without real email delivery.
- D-10..D-18: Implement separate team and project role boundaries.
- D-25, D-26, D-29, D-30, D-31: Persist fixed default board columns mapped to task statuses.
- D-37..D-40: Enforce minimum administrator and project manager constraints in backend services.
</must_haves>
