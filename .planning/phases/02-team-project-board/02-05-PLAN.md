# Plan 05 — Cold Start Migration Guard

<objective>
Close the Phase 2 UAT blocker where a backend started against an old local SQLite schema served `/api/v1/teams` and failed with `sqlite3.OperationalError: no such table: teams`.
</objective>

<user_story>
As a TTCS developer or evaluator, I want the backend to clearly require the latest database migration before it serves Phase 2 APIs, so that a cold-started demo fails early with a useful fix instead of producing runtime 500s.
</user_story>

<gap_source>
`.planning/phases/02-team-project-board/02-UAT.md`

- Test: `1. Cold Start Smoke Test`
- Failed truth: `Cold start applies or requires Phase 2 schema before team/project APIs are used`
- Root cause: `backend/ttcs.db` was still at Alembic revision `20260517_0001`, while Phase 2 requires `20260517_0002`.
</gap_source>

<threat_model>
T-02-GAP-SILENT-SCHEMA: Backend can start with an old database and expose Phase 2 routes that fail at request time.
T-02-GAP-MISLEADING-DOCS: Setup docs can mention migrations but still leave the remediation unclear when startup fails.
T-02-GAP-TEST-DRIFT: A startup guard can break isolated unit tests that intentionally use in-memory SQLite.
T-02-GAP-PROD-MUTATION: Automatically mutating production databases at app startup can be unsafe. Prefer fail-fast readiness by default; make any auto-migration behavior explicit if introduced.
</threat_model>

<must_haves>
- Add a migration readiness check that compares the runtime database Alembic revision to the configured Alembic head.
- Wire the readiness check into backend startup before team/project routes can serve traffic.
- Preserve in-memory SQLite test workflows without requiring Alembic migration setup.
- Provide a clear error message that includes `cd backend && uv run alembic upgrade head`.
- Update backend docs so Phase 2 local startup and cold-start remediation are obvious.
</must_haves>

<task id="02-05-01" title="Add migration readiness helper">
<read_first>
- `backend/alembic.ini`
- `backend/alembic/env.py`
- `backend/app/core/config.py`
- `backend/app/db/session.py`
- `backend/alembic/versions/20260517_0002_create_team_project_board.py`
</read_first>
<action>
- Create `backend/app/scripts/ensure_migrations.py`.
- Implement a helper that reads the current database revision through Alembic and compares it with the migration script head.
- Treat an absent, unknown, or behind revision as not ready.
- Skip the guard for in-memory SQLite URLs used by unit tests.
- Expose a small CLI entry point so developers can run the check directly.
- The failure message must name the current revision when available, the expected head, and the remediation command `cd backend && uv run alembic upgrade head`.
</action>
<verify>
<automated>`cd backend && uv run python -m app.scripts.ensure_migrations`</automated>
</verify>
<acceptance_criteria>
- The helper returns success for a database at Alembic head.
- The helper raises or exits nonzero for a stale file-backed database.
- The helper skips `sqlite:///:memory:` or equivalent in-memory test URLs.
- Failure output is actionable without requiring the developer to inspect backend logs.
</acceptance_criteria>
</task>
