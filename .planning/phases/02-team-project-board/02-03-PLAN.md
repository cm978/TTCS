# Plan 03 — Frontend Data Stores and Routing

<objective>
Create Phase 2 frontend typed API modules, Pinia stores, protected routes, app-shell navigation, and the `/app` team/project start view.
</objective>

<user_story>
As a logged-in TTCS user, I want `/app` to guide me into teams, invitations, and projects, so that I can reach the real project board workspace without fake dashboard data.
</user_story>

<threat_model>
T-02-FE-DATA: Frontend state could show stale or unauthorized data after a `401`. Keep Axios unauthorized handling and stores aligned with auth logout behavior.
T-02-FE-FAKE: UI could invent task/report data before backend support exists. Restrict `/app` and project cards to real team/project/invitation API responses.
T-02-FE-ROUTE: Deep links could bypass auth. All Phase 2 routes must use existing `requiresAuth` guard.
</threat_model>

<must_haves>
- D-19..D-24: Replace Phase 1 foundation entry with real team/project routing and selection behavior.
- D-27, D-28: Do not fake task cards; use real empty state copy.
- D-32, D-34: Route structure supports member management surfaces.
</must_haves>
