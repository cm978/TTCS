# Plan 04 — Integration Hardening and Phase Verification

<objective>
Close Phase 1 by proving the full local register/login/me/logout loop, documenting startup commands, filling test gaps, and recording implementation evidence for verification.
</objective>

<user_story>
As a developer or course reviewer, I want one documented local run path that proves TTCS authentication works end-to-end across frontend, backend, database, and cache.
</user_story>

<threat_model>
T-01-GAP: Unit tests pass but the local full-stack path is broken. Require documented manual smoke across Docker, backend, and frontend.
T-01-DOC: Startup docs omit env or migration steps. Verify docs from a fresh shell where practical.
T-01-LEAK: Debug output exposes tokens or password data. Inspect auth logs/responses and keep token display out of UI except implicit session behavior.
T-01-SCOPE: Final hardening sneaks in team/project/task/dashboard features. Keep final UI and docs scoped to auth foundation only.
</threat_model>

<must_haves>
- D-01: Final smoke must prove register, login, `/me`, logout, and protected route access.
- D-02: Email verification remains out of scope.
- D-03: Password reset remains out of scope.
- D-04: Auth UI remains focused on Phase 1 auth capabilities.
- D-05: Final smoke proves protected access, refresh recovery, and logout redirect/blocking.
- D-06: Refresh tokens, multi-device sessions, and advanced revocation remain out of scope.
- D-09: Documentation states token storage choice and rationale.
- D-16: Startup commands are clear for Docker, backend, and frontend.
- D-18: Final seed/demo check confirms only a user is seeded.
- D-20: Test evidence confirms tests do not rely on dev seed data.
- D-23: Protected content remains a Phase 1 foundation page.
- D-24: No workspace queues or dashboard metrics are implemented.
- D-29: No decorative or fake-data visual shortcuts are introduced.
- D-31: Branded auth screens do not imply teams/projects/tasks are implemented.
- D-32: Final UI check covers visible labels, inline errors, loading/disabled states, focus states, and keyboard controls.
</must_haves>
