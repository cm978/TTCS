# Plan 03 — Auth Frontend Vertical Slice

<objective>
Implement the TTCS login/register/protected-shell experience against the auth backend, following `01-UI-SPEC.md` and `design-system/MASTER.md`.
</objective>

<user_story>
As a TTCS user, I want to create an account, log in, see a protected application shell, refresh without losing my session, and log out cleanly.
</user_story>

<threat_model>
T-01-TOKEN: Browser token persists longer than intended or remains after logout. Store only the JWT, clear on logout and `401`, and avoid storing profile secrets.
T-01-UIERR: Backend errors expose raw stack traces or secrets. Map errors to safe Chinese recovery copy.
T-01-ROUTE: Protected content briefly renders before auth check. Gate routes through auth store hydration before protected page render.
T-01-A11Y: Auth forms become inaccessible. Use visible labels, focus states, keyboard-friendly controls, and live error messaging where practical.
</threat_model>

<must_haves>
- D-01: Implement registration, login, current-user display, logout, and protected route access.
- D-03: Do not implement password reset or password reset placeholder pages.
- D-04: Auth UI stays focused on registration, login, protected access, current-user display, and logout.
- D-05: Browser refresh must recover the logged-in session and logout must block protected pages.
- D-07: Store the Phase 1 bearer token in `localStorage` for MVP/demo ergonomics.
- D-12: Frontend foundation uses Vue 3, Vite, and TypeScript.
- D-21: After login, land in a basic TTCS `AppLayout`, not a blank protected page.
- D-22: Layout includes reusable navigation, current-user affordance, and logout access.
- D-23: Content remains Phase 1 scoped: protected foundation page, not real personal workspace.
- D-24: Do not implement Phase 5 workspace queues or dashboard metrics.
- D-25: Follow UI/UX Pro Max accessibility, interaction feedback, performance, responsive layout, semantic color, form feedback, and navigation principles.
- D-26: Use `design-system/MASTER.md` as visual source of truth.
- D-27: Frontend should feel like a polished operational SaaS cockpit.
- D-28: Do not use emoji as structural icons; prefer `lucide-vue-next`.
- D-29: Do not use decorative gradient orbs, bokeh blobs, fake dashboard data, or marketing-style hero composition.
- D-30: Theme/wrap Ant Design Vue with TTCS tokens.
- D-31: Auth screens should feel branded and product-specific without implying later features exist.
- D-32: Every auth form includes visible labels, inline errors, loading/disabled submit states, accessible focus states, and keyboard-friendly controls.
</must_haves>
