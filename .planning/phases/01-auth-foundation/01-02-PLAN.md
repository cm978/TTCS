# Plan 02 — Auth Backend Vertical Slice

<objective>
Implement the backend user registration, login, current-user, password hashing, JWT, and dev seed slice on top of the skeleton from Plan 01.
</objective>

<user_story>
As a TTCS user, I want to register and log in with email/password, so that I can receive a JWT and access my own account information through a protected API.
</user_story>

<threat_model>
T-01-PWD: Plaintext passwords leak through storage, responses, logs, or tests. Hash with bcrypt and never serialize password fields.
T-01-JWT: Tokens are forged, expired, or signed with weak defaults. Use environment secret, expiry, and explicit algorithm.
T-01-AUTHZ: Protected endpoints accidentally accept missing/invalid tokens. Centralize current-user dependency and test failure cases.
T-01-SEED: Demo credentials become production secrets. Keep seed local/dev-only and configurable.
</threat_model>

<must_haves>
- D-01: Implement the minimal closed auth loop: register, login, `/me`, logout-compatible behavior, and protected access.
- D-02: Do not implement real email verification in Phase 1.
- D-05: Prove authenticated access, browser refresh recovery, and logout invalidating frontend protected access.
- D-06: Do not implement refresh tokens, multi-device session management, or advanced revocation.
- D-07: Choose and document a token storage strategy.
- D-08: Use the 24-hour JWT lifetime default unless implementation finds a better local-demo value.
- D-09: Explicitly state the token storage choice and rationale.
- D-11: Backend foundation uses FastAPI and SQLAlchemy.
- D-13: Alembic migrations exist from the start.
- D-17: Provide a repeatable seed command for one demo user.
- D-18: Seed only a user; do not seed teams, projects, tasks, work logs, or acceptance records.
- D-19: Demo credentials must be local/dev-only and not production secrets.
- D-20: Tests use independent fixtures and must not depend on development seed data.
</must_haves>
