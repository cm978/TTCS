## 1. 设计背景

TTCS 当前需求已覆盖用户、团队、项目、任务、工作日志、阻塞、报表、个人工作台、文档、代码管理和代码审核等模块。但现有任务状态设计主要围绕“创建、流转、完成”，缺少对“任务为什么算完成”的正式判断机制。

如果成员可以直接把任务改为“已完成”，系统就只能证明“有人点了完成按钮”，不能证明任务已经被验收、交付物可用、阻塞已解除、代码或文档证据齐全。这会导致项目 Demo 看起来像任务看板，而不是能支撑真实团队协作的交付系统。

本设计将任务完成机制升级为“证据门禁 + 人工验收”闭环：

```text
个人工作台看到今日任务
  -> 项目看板推进任务
  -> 记录工作日志或阻塞
  -> 满足客观验收条件
  -> 提交待验收
  -> Owner/项目经理 Review
  -> 通过进入已完成，打回则回到进行中
```

## 2. 设计目标

### 2.1 业务目标

- 让 `已完成` 具备可追溯证据，而不是主观状态按钮。
- 支持每日站会、个人执行、阻塞处理、任务提交、项目经理验收的真实工作节奏。
- 将个人工作台从静态统计页升级为行动入口，展示“今日待办、待写日志、待我验收、被打回、阻塞中”等队列。
- 为代码任务后续接入 AI/Agent 辅助评审预留数据和流程边界。

### 2.2 技术目标

- 在第一版中实现清晰可落地的状态机和验收规则，不引入完整规则引擎。
- 验收门禁按任务类型区分，避免所有任务共用同一套粗糙规则。
- 验收流程必须可审计，保留提交、通过、打回、补充证据等历史记录。
- AI/Agent 评审仅作为二期辅助能力，不拥有最终完成权。

## 3. 设计原则

| 原则 | 说明 |
|------|------|
| 证据优先 | 任务进入“待验收”前必须先满足客观证据条件 |
| 人工确认 | 最终完成权归 Owner 或项目经理，不交给自动流程 |
| 分类型验收 | 普通任务、文档任务、代码任务使用不同验收门禁 |
| 阻塞独立 | 阻塞是风险标记，不作为主状态，避免状态机膨胀 |
| 一期收敛 | 一期不做完整规则引擎，不接真实 AI/Agent 评审 |
| 二期可扩展 | 数据模型为 AI/Agent 评审报告、风险项和建议预留扩展点 |

## 4. 核心概念

| 概念 | 定义 |
|------|------|
| 任务类型 | 用于决定验收门禁的任务分类，包括普通任务、文档任务、代码任务 |
| 验收门禁 | 系统自动检查的客观条件，满足后才允许提交验收 |
| 验收证据 | 子任务、工作日志、附件、文档、Commit、PR/MR、代码审核记录等 |
| 验收 Review | Owner 或项目经理对任务完成质量进行人工确认 |
| 打回修改 | 验收未通过时的状态和历史记录，必须包含打回原因 |
| AI/Agent 辅助评审 | 二期能力，针对代码任务输出摘要、风险、测试建议和验收匹配度 |

## 5. 任务状态机

### 5.1 主状态

| 状态 | 说明 | 可操作角色 |
|------|------|-----------|
| 待办 | 任务已创建但尚未开始 | Owner、项目经理、任务参与者 |
| 进行中 | 任务正在执行 | 任务参与者 |
| 待验收 | 任务已满足门禁条件，等待人工 Review | Owner、项目经理 |
| 打回修改 | Review 未通过，需要修改后重新提交 | 任务参与者 |
| 已完成 | Review 通过，任务完成 | Owner、项目经理 |
| 已关闭 | 任务归档或项目收尾后关闭 | Owner、项目经理 |

### 5.2 状态流转

```text
待办 -> 进行中 -> 待验收 -> 已完成 -> 已关闭
                  ^      |
                  |      v
               打回修改 <-
```

### 5.3 流转规则

| 流转 | 触发动作 | 前置条件 | 后置动作 |
|------|---------|---------|---------|
| 待办 -> 进行中 | 开始任务 | 用户为任务参与者 | 记录活动日志 |
| 进行中 -> 待验收 | 提交验收 | 验收门禁全部通过 | 创建验收提交记录，通知 Owner/项目经理 |
| 打回修改 -> 待验收 | 重新提交验收 | 验收门禁全部通过 | 更新验收提交记录，通知验收人 |
| 待验收 -> 已完成 | 通过验收 | 当前用户为 Owner 或项目经理 | 记录验收结果，通知参与者 |
| 待验收 -> 打回修改 | 打回任务 | 当前用户为 Owner 或项目经理，填写打回原因 | 记录打回原因，通知参与者 |
| 已完成 -> 已关闭 | 关闭任务 | 项目收尾或 Owner 手动关闭 | 记录关闭时间 |

### 5.4 阻塞标记

阻塞不作为主状态，而是任务风险标记。

- 任务可以处于 `进行中` 且 `is_blocked=true`。
- 阻塞原因必须不少于 10 字符。
- 存在未解除阻塞时，不允许提交验收。
- 阻塞任务进入每日站会阻塞报告。

## 6. 分类型验收门禁

### 6.1 任务类型

| 类型 | 编码 | 说明 |
|------|------|------|
| 普通任务 | `GENERAL` | 默认类型，适用于需求、调研、配置、沟通等一般工作 |
| 文档任务 | `DOCUMENT` | 需要提交文档、附件或归档材料的任务 |
| 代码任务 | `CODE` | 需要关联 Commit、PR/MR 或代码审核记录的开发任务 |

### 6.2 普通任务门禁

普通任务提交验收前必须满足：

- 子任务全部完成；若没有子任务，则必须填写验收说明。
- 至少存在一条有效工作日志。
- 不存在未解除阻塞。
- 当前用户是任务参与者或 Owner。
- 验收说明不少于 10 字符。

### 6.3 文档任务门禁

文档任务必须满足普通任务门禁，并额外满足：

- 至少上传或关联一个交付文档。
- 文档未被软删除。
- 文档对 Owner 或项目经理可访问。

### 6.4 代码任务门禁

代码任务必须满足普通任务门禁，并额外满足：

- 至少关联一个 Commit、PR/MR 或代码审核记录。
- 如果关联代码审核记录，未解决评论数量必须为 0。
- 如果 PR/MR 已同步状态，则其状态不得为 rejected/closed without merge。
- 提交验收时必须填写功能自测说明或变更摘要。

### 6.5 门禁失败反馈

系统应返回结构化失败原因，便于前端展示待补项：

```json
{
  "can_submit": false,
  "failed_checks": [
    {
      "code": "WORK_LOG_REQUIRED",
      "message": "至少需要 1 条有效工作日志"
    },
    {
      "code": "BLOCKER_UNRESOLVED",
      "message": "存在未解除阻塞，不能提交验收"
    }
  ]
}
```

## 7. 人工验收 Review

### 7.1 验收角色

| 角色 | 权限 |
|------|------|
| 任务参与者 | 可提交验收、重新提交、补充证据，不能直接完成任务 |
| 任务 Owner | 可通过验收、打回任务、要求补充证据 |
| 项目经理 | 可通过验收、打回任务、要求补充证据 |
| 团队管理员 | 默认不介入业务验收，除非同时具备项目角色 |

### 7.2 Review 结果

| 结果 | 说明 | 状态变化 |
|------|------|---------|
| 通过 | 任务符合验收标准 | 待验收 -> 已完成 |
| 打回 | 任务不符合验收标准 | 待验收 -> 打回修改 |
| 要求补充证据 | 缺少日志、附件、PR 或说明 | 状态保持待验收 |

### 7.3 Review 记录要求

每次 Review 必须记录：

- 任务 ID
- Review 人
- Review 结果
- Review 说明
- 打回原因或补充要求
- Review 时间
- 关联验收提交批次

## 9. API 设计

### 9.1 查询验收门禁

```http
GET /api/v1/tasks/{task_id}/acceptance-gate
```

返回任务是否可提交验收，以及失败检查项。

### 9.2 提交验收

```http
POST /api/v1/tasks/{task_id}/acceptance-submissions
```

请求体：

```json
{
  "summary": "登录 API 已完成联调，子任务全部关闭，已关联 PR #18。"
}
```

### 9.3 查询验收历史

```http
GET /api/v1/tasks/{task_id}/acceptance-submissions
```

返回提交记录、Review 记录和证据快照。

### 9.4 Review 验收

```http
POST /api/v1/tasks/{task_id}/acceptance-reviews
```

请求体：

```json
{
  "submission_id": 1001,
  "result": "APPROVED",
  "comment": "功能符合验收标准，允许完成。"
}
```

`result` 可选值：

- `APPROVED`
- `REJECTED`
- `NEED_EVIDENCE`

### 9.5 查询任务证据

```http
GET /api/v1/tasks/{task_id}/acceptance-evidence
```

用于任务详情页展示验收证据区。

### 9.6 二期 AI/Agent 评审

```http
POST /api/v1/tasks/{task_id}/ai-review-reports
GET /api/v1/tasks/{task_id}/ai-review-reports/latest
```

一期仅预留数据结构和入口，不实现真实调用。

## 10. 前端交互设计

### 10.1 任务详情页

任务详情页新增“验收”区域：

- 当前验收状态
- 门禁检查列表
- 验收证据列表
- 提交验收按钮
- 验收历史
- Review 操作区

### 10.2 个人工作台

个人工作台新增行动队列：

- 今日待办
- 待写工作日志
- 阻塞中任务
- 待我提交验收
- 待我验收
- 被打回任务

### 10.3 项目看板

看板列建议包含：

- 待办
- 进行中
- 待验收
- 打回修改
- 已完成

任务卡片应展示：

- 任务类型
- 验收状态
- 阻塞标记
- 参与者
- 待补证据数量
- 打回次数

### 10.4 Review 弹窗

Review 弹窗展示：

- 验收说明
- 门禁检查结果
- 证据列表
- 工作日志摘要
- 代码任务的 PR/Commit/审核记录
- 操作按钮：通过、打回、要求补充证据

## 11. 与现有模块的关系

| 模块 | 影响 |
|------|------|
| 任务管理 | 新增任务类型、验收状态、验收提交、打回流程 |
| 工作日志 | 成为普通任务和代码任务的基础验收证据 |
| 阻塞机制 | 未解除阻塞会阻止提交验收 |
| 文档管理 | 文档任务需要关联交付文档 |
| 代码管理 | 代码任务需要关联 Commit、PR/MR 或代码审核记录 |
| 代码审核 | 未解决评论会阻止代码任务提交验收 |
| 个人工作台 | 新增待验收、待我验收、被打回等行动队列 |
| 报表统计 | 可统计验收通过率、打回次数、平均验收时长 |
| 通知中心 | 提交验收、通过、打回、补充证据都触发通知 |

## 12. 通知设计

| 事件 | 接收人 | 通知内容 |
|------|--------|----------|
| 提交验收 | Owner、项目经理 | 任务已提交验收，请 Review |
| 验收通过 | 任务参与者 | 任务验收通过，状态已完成 |
| 验收打回 | 任务参与者 | 任务被打回，需查看原因并修改 |
| 要求补充证据 | 提交人 | 验收证据不足，请补充 |
| 阻塞未解除 | 任务参与者、Owner | 存在未解除阻塞，不能提交验收 |

## 13. 测试要点

| 测试场景 | 预期结果 |
|----------|---------|
| 普通任务无工作日志时提交验收 | 提交失败，提示缺少工作日志 |
| 存在未解除阻塞时提交验收 | 提交失败，提示先解除阻塞 |
| 文档任务无交付文档时提交验收 | 提交失败，提示缺少文档证据 |
| 代码任务无 PR/Commit 时提交验收 | 提交失败，提示缺少代码证据 |
| 非 Owner/项目经理 Review | 返回 403 权限错误 |
| Owner 打回任务但未填写原因 | 提交失败，提示必须填写打回原因 |
| 验收通过后查看任务详情 | 显示验收人、时间、说明和证据快照 |
| 被打回任务重新提交 | 创建新的提交记录，保留历史 Review |

## 15. 决策结论

TTCS 第一版采用“证据门禁验收闭环”：

- 不再允许成员直接把任务置为已完成。
- 任务必须先满足分类型客观门禁，再提交验收。
- Owner 或项目经理进行最终 Review。
- 打回、补充证据和验收通过都必须进入历史记录。
- 个人工作台展示真实行动队列，成为每日执行入口。
- AI/Agent 评审作为二期能力，只做辅助 reviewer，不拥有最终完成权。
