# TTCS 项目 Agent 指南

本文档用于初始化 TTCS（Team Task Collaboration System）项目的开发上下文，供后续 AI coding agent、开发者和评审者快速对齐需求、技术栈、工程边界与实现规则。

## 1. 项目定位

TTCS 是一个面向小型创业团队（5-15 人）的团队任务协作管理系统，采用 B/S 架构，提供任务分配、项目看板、协作沟通、报表统计、文档管理、工作日志、代码管理与代码审核等能力。

核心业务目标：

- 提高任务分配效率，减少人工协调成本。
- 提升团队沟通效率，减少无效会议。
- 实时展示项目进度和团队工作状态。
- 积累任务、工时、阻塞和协作数据，用于管理分析。

## 5. 核心功能范围

必须覆盖的业务模块：

- 用户管理：注册、登录、邮箱验证、密码找回、个人信息。
- 团队管理：创建团队、邀请成员、角色分配、成员管理。
- 项目管理：创建项目、项目配置、项目成员、看板配置。
- 任务管理：任务 CRUD、状态流转、优先级、截止时间、子任务、依赖关系。
- 多人协作：任务参与者最多 5 人，任务 Owner 保持责任归属。
- 工作日志：每日工作记录、工时、进度、阻塞上报、Git 字段预留。
- 阻塞机制：阻塞原因不少于 10 字符，通知任务 Owner，看板展示阻塞标记，进入阻塞报表。
- 协作沟通：评论、@提及、活动动态、实时通知、附件上传。
- 个人工作台：我的任务、待办、逾期、项目进度、最近活动。
- 日历视图：日/周/月视图，按任务日期组织展示。
- 文档管理：文件上传、下载、归档、权限控制。
- 侧边栏导航：主导航、项目快捷入口、成员在线状态。
- 成员主页：成员信息、任务、项目、工作日志和权限控制。
- 报表统计：燃尽图、任务分布、工时统计、阻塞汇总、项目进度。
- 代码管理：仓库绑定、提交历史、代码浏览、Diff 对比。
- 代码审核：创建审核、分配审核人、文件级评论、状态流转。
- 系统管理：系统日志、备份、配置管理。

## 6. 关键业务规则

任务与工作日志：

- 任务使用 `owner_id` 表示创建者/责任人，使用参与者关联表表示多人协作。
- 任务参与者上限为 5 人。
- 非任务参与者不能记录该任务的工作日志。
- 任务进度应优先基于子任务完成度客观计算，不依赖主观填报。
- 工作日志支持 `is_blocked` 和 `blocked_reason`。
- 阻塞原因必须有明确描述，最少 10 字符。
- 工作日志预留 `commit_hash`、`branch_name`、`repository_url`、`git_synced` 字段，便于二期 Git Webhook 自动同步。

权限与安全：

- 使用 RBAC 进行权限控制。
- 所有项目、任务、文档、工作日志读取都必须校验团队/项目可见性。
- 密码使用 bcrypt 存储，成本因子 12。
- OAuth 令牌和敏感数据必须加密存储。
- 文件上传需要校验类型、大小和访问权限。
- 删除任务和文件优先使用软删除，并记录操作日志。

通知：

- 实时通知通过 WebSocket 推送。
- WebSocket 心跳间隔为 30 秒。
- 在线状态根据 WebSocket 连接和最近活动判断。
- 通知失败不能阻塞主业务，应记录日志并保留未读状态。

## 7. 非功能指标

性能：

- 首屏加载时间 < 2 秒。
- 95% API 请求响应 < 500ms。
- 工作台数据加载 < 1 秒。
- 日历月视图加载 < 2 秒。
- 成员主页加载 < 1.5 秒。
- Diff 1000 行以内渲染 < 2 秒。
- 支持 100 个并发用户。

可用性与可靠性：

- 月度可用性 >= 99.5%。
- 故障恢复时间 <= 4 小时。
- 每日自动备份。
- 数据恢复时间 <= 2 小时。
- 核心业务事务必须保证一致性。

质量：

- 后端核心模块单元测试覆盖率 >= 70%。
- 关键函数应有必要注释。
- 操作日志保留 180 天。
- 配置通过环境变量管理，禁止硬编码密钥。

## 8. 前端实现原则

- 采用 Ant Design Vue 风格，主色为 `#1890FF`。
- 页面风格应偏向高效、清晰、适合长期使用的协作工具，而不是营销型落地页。
- 常用操作不超过 3 次点击。
- 支持桌面和平板响应式布局，移动端使用简化布局。
- 桌面端顶部导航高度参考 64px，侧边栏宽度参考 256px。
- 任务看板、日历、工作台、成员主页应优先保证信息密度和可扫描性。
- 错误提示必须清晰、可行动，避免只显示“失败”。
- 任务卡片需展示状态、优先级、负责人/参与者、阻塞标记和关键标签。

## 9. 后端实现原则

- API 路径使用 `/api/v1/` 版本前缀。
- API 数据格式统一使用 JSON 和 UTF-8。
- 业务逻辑放在 service 层，路由层只做参数解析、认证和响应组装。
- 数据访问通过 repository 或 ORM 查询封装，避免在路由中散落复杂 SQL。
- 需要事务的业务包括：创建任务及初始参与者、状态流转、工作日志与进度更新、阻塞通知、成员角色变更。
- 复杂列表接口应支持分页、过滤和排序。
- WebSocket 推送应和业务事务解耦，优先保证主数据写入成功。
- 统一错误码和错误响应结构，保留权限、校验、资源不存在、冲突等类型。

- Python 模型类使用 PascalCase，例如 `TaskAssignee`。
- API 字段建议使用 snake_case，与后端和数据库保持一致；前端可通过类型层适配。

## 11. 测试重点

每次实现或修改功能时，优先覆盖：

- 用户注册、登录、JWT 校验和权限拒绝。
- 团队、项目、任务的跨权限访问。
- 任务参与者添加/移除，尤其是 5 人限制。
- 工作日志创建、工时范围、日期范围和非参与者禁止记录。
- 阻塞标记、通知发送、看板标记和阻塞报表。
- 子任务完成度对任务进度的自动计算。
- 文件上传权限和软删除。
- GitHub OAuth 令牌加密存储与最小 scope。
- Diff 大文件或大行数的性能边界。

## 12. 开发工作流

建议后续 agent 按如下顺序推进：

1. 先补齐 `02-design/` 设计文档：概要设计、详细设计、数据库设计、API 设计。
2. 初始化后端 FastAPI 工程，建立配置、数据库连接、认证、基础错误处理。
3. 初始化前端 Vue/Vite 工程，建立路由、布局、状态管理和 API client。
4. 先实现用户、团队、项目、任务四个主链路。
5. 再实现工作日志、阻塞机制、通知和报表。
6. 最后实现文档管理、成员主页、代码管理和代码审核。

任何实现前，应先确认当前需求文档是否有更新；若发现文档之间冲突，按“需求基线”章节的优先级处理，并在提交说明或变更文档中记录取舍。

## 13. Agent 行为规则

后续 AI agent 在本项目中应遵守：

- 默认使用中文沟通和撰写项目文档，代码标识符保持英文。
- 修改需求或设计前，先说明受影响模块和原因。
- 不要在没有需求依据的情况下扩大范围。
- 不要把旧章程中的“代码集成二期”误当成当前范围，当前以 SRS V2.3 为准。
- 不要硬编码密钥、Token、数据库密码或第三方凭证。
- 不要绕过权限校验实现快捷接口。
- 不要把 WebSocket 推送失败作为主业务失败。
- 不要引入与技术栈冲突的框架，除非先更新设计决策。
- 对数据库结构、API 契约、权限模型的修改必须同步更新设计文档。
- 完成功能后至少运行对应 lint/test；如果无法运行，必须说明原因。
